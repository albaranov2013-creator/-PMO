Шаблон чек-листа актуальности готовности банка к запуску

Рабочий инструмент на 1-5 рабочих дней до старта. Чек-лист не заменяет протоколы тестирования и решение go / no-go, а проверяет, что ранее сданные результаты и ранее принятое решение о запуске по-прежнему актуальны на дату старта.

Контур: ПСИ → запуск Заполняется с банком Проверяет актуальность, а не факт прошлых тестов Ловит скрытые блокеры вне TS

Правила использования

Как читать статусы и когда чек-лист считается непройденным

Правила заполнения
  • по каждому пункту должен быть владелец со стороны банка;
  • статус Да означает подтверждено и актуально на дату проверки;
  • статус Нет означает блокер или высокий риск;
  • статус Частично означает необходимость решения Go with conditions.
Когда чек-лист не пройден
  • есть хотя бы один статус Нет в ключевых блоках;
  • по внутренним замечаниям банка нет прозрачности;
  • после тестирования были изменения без повторной проверки;
  • банк не готов письменно подтвердить актуальность результатов тестирования.
Да — подтверждено и актуально Частично — запуск возможен только с условиями Нет — блокер или высокий риск

Вывод менеджера

Итоговое заключение по запуску

Краткий вывод
  • Результаты тестирования на дату запуска: актуальны / актуальны с условиями / неактуальны
  • Внутренние блокеры банка: отсутствуют / есть
  • Изменения после тестирования: отсутствуют / есть контролируемые / есть неконтролируемые
  • Рекомендация менеджера: Go / Go with conditions / No-Go
Комментарий менеджера

Короткий текст на 5-10 строк: что подтверждено, какие есть риски, почему запуск рекомендуется или не рекомендуется.

Короткая форма управленческого вывода

По состоянию на дату проверки банк подтверждает или не подтверждает, что ранее согласованная готовность к запуску остается актуальной.

Правило 1

Если есть хотя бы один статус Нет в ключевых блоках, запуск выносится на отдельное решение No-Go или Go with conditions.

Правило 2

Если протоколы положительные, но по внутренним замечаниям банка нет прозрачности, чек-лист не считается пройденным.

Правило 3

Если после тестирования были изменения без повторной проверки, запуск считается находящимся в зоне повышенного риска.
