Типовые риск-паттерны клиента

Повторяющиеся риски

  • частая смена ответственных со стороны заказчика;
  • отсутствие полноценной ИТ-команды;
  • недостаток профильных аналитиков;
  • низкое качество исходных данных для миграции;
  • задержки критичных внешних зависимостей;
  • разрыв между проектной документацией и фактическим состоянием продуктовой ветки;
  • проблемы приемки регуляторной отчетности.

Сработавшие риски по завершенным проектам

  • частая смена ответственных со стороны заказчика;
  • отсутствие ИТ-команды на стороне клиента;
  • недостаточная готовность проектной команды клиента;
  • низкое качество исходных данных для миграции;
  • задержка получения ключей Informatica;
  • международные коммуникационные барьеры;
  • задержка приемки регуляторной отчетности;
  • неподготовленность среды / некорректный перенос настроек;
  • разрыв между продуктовой веткой и проектной документацией.

Триггеры повышенного риска

  • запуск проекта при неготовой команде клиента;
  • отсутствие формализованной продуктовой линейки и требований;
  • последовательные проверки и медленные согласования;
  • зависимость от международных согласований;
  • неготовность исходных данных и инфраструктуры к миграции;
  • межмодульные инциденты, требующие координации нескольких ответственных заказчика.

Критерии приемки

Что обязательно включать в открывающую справку

  • риски по зрелости команды клиента;
  • риски по качеству исходных данных;
  • риски по готовности инфраструктуры;
  • риски по международным коммуникациям и согласованиям;
  • риски по регуляторной отчетности;
  • явные границы между внедрением АБС и работами по организационной трансформации клиента;
  • ограничения состава решения по этапам и модулям.

Рекомендации PMO по управлению клиентом

Какого уровня менеджера назначатьсильный менеджер внедрения
Нужен ли ведущий менеджер
Нужен ли помощник менеджера
Нужен ли усиленный контроль руководителя портфеляда

Обязательные контрольные точки

  • готовность команды клиента;
  • готовность среды;
  • качество исходных данных до старта миграции;
  • статус внешних лицензий;
  • подтверждение логики регуляторной отчетности;
  • фиксация отклонений по срокам со стороны клиента;
  • актуальная версия базового плана проекта;
  • актуальная версия почасового плана перехода.

Какие ошибки нельзя повторять

  • открывать проект без оценки зрелости клиента;
  • начинать миграцию без измеримого порога качества данных;
  • запускать проект при неготовой инфраструктуре;
  • оставлять невыравненной продуктовую ветку;
  • размывать границы проектного скоупа.

Какие лучшие практики уже сработали

  • ограничение кастомизации;
  • фокус на стандартной функциональности;
  • сильные управленческие решения по рамкам проекта;
  • новый подход к сбору метрик по миграции;
  • формализация взаимодействия в TS;
  • отдельный комплект проектной документации по коммуникациям, средам, ПСИ и архитектуре.
